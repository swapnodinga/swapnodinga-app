3. Step 1: State Management (The Brain)

This is the core of your app. All data flows through here.

File: client/src/context/SocietyContext.tsx

What This Does:

  • Defines the structure of your data (Types)
  • Stores member list, transactions, and current user
  • Provides functions to add/update/delete data
  • Simulates API calls (in a real app, these would call a backend)

Key Code Sections:

1. Define Types (Data Shape)

export interface Member { id: string; name: string; email: string; role: "admin" | "member"; status: "active" | "pending"; monthlyInstalment: number; fixedDeposit: number; totalInstalmentPaid: number; totalInterestEarned: number; joinDate: string; }

2. Create Context

const SocietyContext = createContext<SocietyContextType | undefined>(undefined);

3. Create Provider Component

export function SocietyProvider({ children }) { const [members, setMembers] = useState<Member[]>(MOCK_MEMBERS); const [currentUser, setCurrentUser] = useState<Member | null>(null); const login = async (email: string, pass: string) => { const user = members.find(m => m.email === email); if (user) { setCurrentUser(user); return true; } return false; }; return ( <SocietyContext.Provider value={{ members, currentUser, login }}> {children} </SocietyContext.Provider> ); }

4. Create Custom Hook

export function useSociety() { const context = useContext(SocietyContext); if (!context) { throw new Error("useSociety must be used within SocietyProvider"); } return context; }

‚úÖ Why This Pattern?

The custom hook (useSociety) makes it easy to access data from any component. Instead of importing Context directly, components use: const { members, login } = useSociety();

8. Step 6: Routing & Final Assembly

Connect everything together in App.tsx.

File: client/src/App.tsx

How Routing Works:

import { Switch, Route } from "wouter"; function Router() { return ( <Switch> <Route path="/" component={LandingPage} /> <Route path="/admin" component={AdminDashboard} /> <Route path="/dashboard" component={MemberDashboard} /> <Route path="/about" component={AboutPage} /> <Route path="/contact" component={ContactPage} /> <Route component={NotFound} /> </Switch> ); } function App() { return ( <QueryClientProvider client={queryClient}> <TooltipProvider> <SocietyProvider> <Toaster /> <Router /> </SocietyProvider> </TooltipProvider> </QueryClientProvider> ); }

Provider Hierarchy (Nesting):

  1. QueryClientProvider - For data fetching (React Query)
  2. TooltipProvider - For UI tooltips
  3. SocietyProvider - Our custom global state (members, auth)
  4. Toaster - For notifications
  5. Router - Page routing

üí° Why Nested Providers?

Each provider wraps the components below it. This allows all child components to access data from any provider. For example, any page can use useSociety() because they're inside the SocietyProvider.

9. Learning Tips & Best Practices

üéì Understanding the Data Flow

  1. User logs in ‚Üí login() in SocietyContext
  2. setCurrentUser() updates the state
  3. Any component using useSociety() sees the new user
  4. Components re-render with new data automatically

üì¶ Component Reusability

Notice how TransactionTable is used in both Admin and Member dashboards? This is React's power:

  • Write once, use everywhere
  • Pass different data and callbacks via props
  • Admin version shows "Approve/Reject" buttons
  • Member version doesn't
